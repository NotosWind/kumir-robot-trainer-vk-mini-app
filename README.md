# КуМир-Робот — тренажёр для ОГЭ (задание 15)

VK Mini App, который в игровом формате обучает школьников решать **задание 15 ОГЭ по информатике** — написание алгоритма для исполнителя **Робот** на языке КуМир. Язык КуМир не реализуется целиком: делается точная **имитация** исполнителя Робот (поле в клетку со стенами, команды и конструкции из школьного подмножества) и автоматическая проверка решений.

Курсовая работа. Главный результат — работающее приложение; этот README + документы в `docs/` служат пояснительной запиской.

## Возможности

- **Имитация исполнителя Робот**: бесконечное (конечное достаточного размера) поле в клетку, стены на рёбрах клеток, команды `вверх/вниз/влево/вправо`, `закрасить`.
- **Полное подмножество языка задания 15**: цикл `нц пока <условие> … кц`, цикл `нц <N> раз … кц`, ветвление `если … то … иначе … все`; предикаты `слева/справа/сверху/снизу свободно`/`… стена`, `клетка закрашена`/`клетка чиста`; логические связки `не`/`и`/`или`.
- **Гибридный редактор**: вкладка **«Блоки»** — конструктор из атомарных блоков «по словам» (как в настоящем КуМире); вкладка **«Код»** — тот же алгоритм как редактируемый КуМир-текст. Источник правды — общий AST, поэтому блоки и текст всегда синхронны.
- **Пошаговое исполнение и анимация**: «Запустить», «Шаг», «Сброс»; визуальный разбор ошибок (Робот врезался в стену, зациклился), понятные сообщения для 9-классника (сколько клеток не хватает / лишние).
- **Мульти-полевая автопроверка** — как в реальном ОГЭ: решение должно работать на **нескольких полях-вариантах**, а не на одном. Это тренирует главный навык — обобщение.
- **Кампания из 14 уровней**: онбординг (по одному понятию за раз, с урезанной палитрой) → нарастающая сложность (циклы, стены, два цикла, ветвления). Звёзды за качество (правильность / обобщение / эффективность), карта с замками.
- **Сохранение прогресса**: офлайн-first — мгновенно в `localStorage`, с фоновой синхронизацией на бэкенд, когда он доступен.

## Скриншот сценария

```
Карта кампании  →  Уровень (поле + Блоки/Код + Запустить)  →  Результат (звёзды / разбор ошибки)
```

На главном экране уровня слева — SVG-поле с Роботом, закрашенными и целевыми (заштрихованными) клетками и стенами; справа — гибридный редактор; снизу — кнопки запуска. На широком экране поле и редактор показываются рядом (адаптив mobile-first).

## Технологии

**Фронтенд (`frontend/`)**
- React 18 + TypeScript
- Vite 8 (сборка/дев-сервер)
- VKUI 7, `@vkontakte/vk-bridge` (v2), `@vkontakte/vk-mini-apps-router`

**Бэкенд (`backend/`)**
- Python 3.12 + FastAPI
- SQLAlchemy 2 + SQLite (в проде заменяется на PostgreSQL через `DATABASE_URL`)
- Uvicorn (ASGI-сервер)
- Тесты: pytest + httpx (`TestClient`)

## Быстрый старт

Нужны два процесса: дев-сервер фронтенда и (опционально) бэкенд. Приложение работает и **без** бэкенда — прогресс тогда хранится только в `localStorage`.

### 1. Фронтенд

```bash
cd frontend
npm install
npm run dev
```

Открыть **http://127.0.0.1:5174/** (именно `127.0.0.1`, а не `localhost` — см. нюансы). Хост и порт заданы в `frontend/vite.config.ts`, поэтому дополнительные флаги не нужны.

### 2. Бэкенд (опционально)

```bash
cd backend
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt          # macOS/Linux: .venv/bin/pip install -r requirements.txt

# Локальная разработка без VK (авторизация вернёт тестового пользователя):
set VK_SKIP_AUTH=1                                      # macOS/Linux: export VK_SKIP_AUTH=1
.venv\Scripts\python -m uvicorn app.main:app --reload --port 8000
```

Дев-сервер фронтенда **проксирует** `/api` → `http://localhost:8000` (см. `frontend/vite.config.ts`), поэтому фронт и бэкенд работают вместе без настройки CORS.

### 3. Продакшен-сборка фронтенда

```bash
cd frontend
npm run build        # tsc -b && vite build → dist/
npm run preview      # локальный предпросмотр собранного приложения
```

## Тестирование

### Фронтенд (Vitest)

```bash
cd frontend
npm test             # весь набор (~112 тестов)
npm run test:watch   # watch-режим
npx vitest run src/engine            # только тесты движка
npx tsc -b           # проверка типов (Vitest сам типы НЕ проверяет — гонять отдельно!)
```

Что покрыто:
- **движок** — каждая команда, столкновения со стенами, семантика циклов и ветвлений, предикаты, защита от зацикливания, расчёт звёзд, round-trip «текст ↔ AST» и «блоки ↔ AST»;
- **логика состояния** — редьюсер сессии, воспроизведение анимации по кадрам, синхронизация блоки↔код, прогресс, навигация;
- **компоненты** — поле, редакторы, карта, модалка результата (включая сообщения об ошибках);
- **интеграция** — полный цикл «открыть уровень → решить → звёзды → возврат на карту → разблокировка следующего».
- **все 14 уровней** валидируются эталонным решением: тест прогоняет решение через движок на всех полях уровня.

### Бэкенд (pytest)

```bash
cd backend
.venv\Scripts\python -m pytest -q     # macOS/Linux: .venv/bin/python -m pytest -q
```

Что покрыто: health-эндпоинт, модель прогресса, проверка подписи VK launch-параметров, эндпоинты прогресса (GET/PUT, «лучший по звёздам», изоляция по пользователю).

## Как устроено ядро (движок и проверка)

**Система координат.** `x` — столбец (0 слева), `y` — строка (0 сверху), `y` растёт **вниз**. Движения: `вверх` → `y−1`, `вниз` → `y+1`, `влево` → `x−1`, `вправо` → `x+1`.

**Стены** хранятся на рёбрах клеток строками-ключами:
- `"V:x:y"` — вертикальная стена справа от клетки `(x, y)` (между `(x,y)` и `(x+1,y)`);
- `"H:x:y"` — горизонтальная стена снизу от клетки `(x, y)` (между `(x,y)` и `(x,y+1)`).

Граница поля всегда считается стеной. `<направление> свободно` ⟺ соседняя клетка в пределах поля и на ребре нет стены; `<направление> стена` — отрицание.

**Крах и зацикливание.** Команда движения в стену — ошибка исполнения: Робот «разрушается», исполнение останавливается со статусом `crash`. Если число выполненных команд превышает лимит — статус `looplimit` (защита от бесконечного цикла).

**Проверка и звёзды.** Решение прогоняется на **каждом** поле уровня; поле зачтено, если статус `ok` и множество закрашенных клеток точно совпадает с целью.
- ⭐ — решено на видимом поле;
- ⭐⭐ — решение проходит **все** поля (обобщает — главный навык ОГЭ);
- ⭐⭐⭐ — все поля **и** эффективно: использован цикл и число команд не превышает порог уровня.

> **Важный методический момент.** Цикл `нц пока` проверяет условие *перед* телом, поэтому `нц пока справа свободно { закрасить; вправо }` оставляет **последнюю** клетку незакрашенной — это и есть классическая «ошибка забора», которой учит задание 15. Приложение явно подсказывает об этом в разборе ошибки.

## Бэкенд и авторизация VK

- **Авторизация.** Фронтенд передаёт «сырую» строку VK launch-параметров (`window.location.search` без `?`) в заголовке `X-VK-Launch-Params`. Бэкенд проверяет подпись `sign` (HMAC-SHA256 по `vk_*`-параметрам с секретом приложения, base64url) и извлекает `vk_user_id`. Паролей нет.
- **Эндпоинты:** `GET /api/health`, `GET /api/progress`, `PUT /api/progress/{level_id}`.
- **Переменные окружения** (см. `backend/.env.example`):
  - `VK_APP_SECRET` — секрет VK-приложения;
  - `DATABASE_URL` — строка подключения (по умолчанию SQLite-файл; для прода, например, PostgreSQL);
  - `VK_SKIP_AUTH=1` — режим локальной разработки без VK;
  - `VK_DEV_USER_ID` — id тестового пользователя в dev-режиме.
- **Прод.** Задать `VK_APP_SECRET` и реальный `DATABASE_URL`, `VK_SKIP_AUTH` не выставлять, запускать `uvicorn app.main:app`.

Подробнее — `backend/README.md`.

## Известные нюансы среды

Проект разрабатывался на **Node.js 22.2.0** (ниже рекомендуемого 22.12), отсюда несколько обходных решений — если у тебя Node ≥ 22.12, большинство из них просто не нужны:

- **Vitest 4 / rolldown.** Платформенный нативный биндинг иногда не ставится (баг npm с optional-зависимостями). Закреплён `@rolldown/binding-win32-x64-msvc` в `optionalDependencies` фронтенда.
- **happy-dom вместо jsdom.** jsdom на этом Node падает на ESM-interop, поэтому тесты компонентов используют happy-dom (`// @vitest-environment happy-dom` в `*.test.tsx`).
- **Дев-сервер — на `127.0.0.1`.** Хост `127.0.0.1` и порт `5174` зашиты в `vite.config.ts` (`server.host`/`server.port`), иначе `localhost` может уйти в IPv6 и инструменты получают connection-refused. В браузере открывать `http://127.0.0.1:5174/`. (Передавать `--host/--port` через `npm run dev --` в PowerShell ненадёжно — npm может «потерять» имена флагов.)
- **vk-bridge под Vite dev.** Дев-пребандлер Vite мог отдавать default-импорт `@vkontakte/vk-bridge` как неймспейс (метод `send` оказывался на `.default`) — был белый экран. Исправлено в `src/app/bridge.ts` (`resolveBridge()` обрабатывает обе формы). Прод-сборка от этого не страдает.
